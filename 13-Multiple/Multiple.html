Overview

Multiple simulates the gravitational motion of any number of point masses interacting under Newtonian gravity in full three dimensions. It is a direct generalisation of the two-body Binary program: where Binary tracks two bodies in the \(xy\)-plane, Multiple handles \(N\) bodies in \((x, y, z)\) space, with every body exerting a gravitational force on every other.

The user specifies the number of bodies, their masses (in solar masses), their three-dimensional initial positions and velocities, and a set of integration-control parameters. The program integrates the coupled equations of motion forward in time using a predictor–corrector algorithm with an adaptive timestep, and produces either a trajectory plot of all bodies projected onto the \(xy\)-plane, or a snapshot of current positions at regular output intervals.

The default configuration demonstrates a classic three-body interaction: three equal-mass stars approach one another, two form a gravitationally bound binary pair, and the third is ejected — a common outcome of stellar encounters that Multiple makes easy to reproduce and explore.

Scope: Multiple uses Newtonian gravity only. No relativistic corrections are applied. For two-body simulations with higher numerical accuracy, see Binary_RK45. Like Binary, Multiple is vulnerable to timestep collapse near close encounters; see the Adaptive Timestep section for guidance. An RK45-based version of Multiple is available at Multiple_RK45.

Quick Start

Requirements

Python 3.9 or later with numpy and matplotlib:

pip install numpy matplotlib

Running the Program

cd GFTGU-Programs/Multiple
python main.py

The default configuration simulates three equal-mass stars (\(M = 1\,M_\odot\) each) starting at positions \((4.6\times10^{10}, 0, 0)\,\mathrm{m}\), \((-4.6\times10^{10}, 0, 0)\,\mathrm{m}\), and \((0, 4.6\times10^{10}, 0)\,\mathrm{m}\), with initial velocities chosen to produce a close three-body encounter. The run lasts 600,000 steps of \(\Delta t = 2{,}000\,\mathrm{s}\), with trajectory output sampled every 1,000 steps.

Customising the Simulation

All parameters are set in the SimulationParams block near the top of main.py. The key fields are:

  • n_bodies — number of bodies
  • masses_solar — list of masses in solar-mass units, one per body
  • positions_init — list of [x, y, z] starting positions in metres
  • velocities_init — list of [vx, vy, vz] starting velocities in m s−1
  • dt — maximum timestep in seconds
  • max_steps — total number of accepted integration steps
  • out_steps — interval between output events (steps)
  • output_type — 'trajectories' or 'current_positions'
  • eps1 — adaptive timestep tolerance
  • eps2 — corrector convergence tolerance

Make sure that masses_solar, positions_init, and velocities_init each contain exactly n_bodies entries. No other file needs editing for typical use.

Memory usage: In 'trajectories' mode, positions are stored at every accepted step, which requires memory proportional to max_steps × n_bodies × 3 floating-point numbers. For large runs with many bodies, use 'current_positions' mode and set out_steps small enough to visualise the evolution without exhausting available memory.

Program Files

main.py
Entry Point & Configuration

Defines all simulation parameters in a single SimulationParams dataclass instance — number of bodies, masses, initial positions and velocities, timestep, step limit, output interval, output type, and both tolerance thresholds. Calls run_simulation() to integrate the trajectory, then dispatches to plot_trajectories() or plot_current_positions() depending on output_type. All user-adjustable settings live here; no other file needs editing for typical use.

physics_multiple.py
Physics Engine

Defines the physical constants \(G = 6.67430\times10^{-11}\,\mathrm{N\,m^2\,kg^{-2}}\), \(M_\odot = 1.98847\times10^{30}\,\mathrm{kg}\), and \(k = G\,M_\odot\) (matching Schutz's kGravity convention). Provides the single function compute_accelerations(positions, masses_solar), which accepts the \(N\times3\) position array and the \(N\)-element mass array (in solar-mass units) and returns the \(N\times3\) acceleration array. Internally it iterates over all \(N(N-1)/2\) distinct pairs using a double loop and applies Newton's third law to populate both bodies' contributions from a single pair computation.

driver_multiple.py
Integration Driver

Defines the SimulationParams dataclass and the main integration function run_simulation(params). The driver advances all \(N\) bodies simultaneously using an Euler predictor followed by an averaged-acceleration corrector, with adaptive timestep halving controlled by eps1 (on predicted accelerations) and eps2 (on corrected positions). In 'trajectories' mode it accumulates all positions into a NumPy array of shape (max_steps, n_bodies, 3) and returns it together with the time array. In 'current_positions' mode it stores only a positional snapshot every out_steps accepted steps, keeping memory use bounded regardless of run length.

plot_multiple.py
Visualisation

Provides two functions. plot_trajectories(result) renders a single matplotlib figure showing the \(xy\)-projection of each body's full trajectory, with bodies coloured in the sequence red, green, blue, orange, purple, brown (cycling for \(N > 6\)). plot_current_positions(result) renders a scatter plot of all bodies' positions at the most recent snapshot, labelled with the simulation time. Both functions use equal aspect ratio and display a legend identifying each body by index.

N-Body Problem

The gravitational N-body problem describes the motion of \(N\) point masses \(m_0, m_1, \ldots, m_{N-1}\) interacting through Newtonian gravity. Each body is attracted toward every other; there are no central forces and no prescribed orbits. Unlike MercPert, where the binary moves on a fixed analytical orbit and only Mercury is integrated, Multiple integrates all bodies on equal footing.

The motion is fully three-dimensional. Each body's state is described by its position vector \(\mathbf{r}_A = (x_A, y_A, z_A)\) and velocity vector \(\dot{\mathbf{r}}_A = (v_A, u_A, w_A)\), where following Schutz's convention \(v\), \(u\), \(w\) denote the \(x\)-, \(y\)-, and \(z\)-components of velocity respectively.

Visualisation: The simulation is three-dimensional but plot_multiple.py projects all trajectories onto the \(xy\)-plane. For a different projection, rotate the coordinate system by transforming the initial conditions, or post-process the stored position arrays to apply any desired viewing direction.

Equations of Motion

The gravitational acceleration on body \(A\) due to body \(B\) is given by Newton's law of universal gravitation, directed from \(A\) toward \(B\):

(1) \[ \mathbf{a}_{A \leftarrow B} \;=\; \frac{G\,m_B}{r_{AB}^3}\,\bigl(\mathbf{r}_B - \mathbf{r}_A\bigr) \]

where \(r_{AB} = |\mathbf{r}_A - \mathbf{r}_B|\) is the scalar separation between the two bodies. The total acceleration on body \(A\) is the vector sum over all other bodies:

(2) \[ \ddot{\mathbf{r}}_A \;=\; \sum_{\substack{B=0 \\ B \neq A}}^{N-1} \frac{G\,m_B}{r_{AB}^3}\,\bigl(\mathbf{r}_B - \mathbf{r}_A\bigr) \]

In the code, masses are stored in solar-mass units and the constant \(k = G\,M_\odot\) is factored out, so the acceleration on \(A\) due to \(B\) is written as:

(3) \[ \mathbf{a}_{A \leftarrow B} \;=\; \frac{k\,\tilde{m}_B}{r_{AB}^3}\,\bigl(\mathbf{r}_B - \mathbf{r}_A\bigr) \]

where \(\tilde{m}_B\) is the mass of body \(B\) in solar-mass units and \(k = G\,M_\odot = 1.327\times10^{20}\,\mathrm{m^3\,s^{-2}}\) (matching Schutz's kGravity convention).

Pair-Loop Acceleration

The key efficiency of the implementation is that the acceleration of body \(A\) on body \(B\) and the acceleration of \(B\) on \(A\) are both computed in a single pass over the pair \((A, B)\), using Newton's third law:

(4) \[ \mathbf{a}_{A \leftarrow B} \;=\; -\,\mathbf{a}_{B \leftarrow A} \]

Define the displacement vector from body \(A\) to body \(B\) and the common geometric factor:

(5) \[ \boldsymbol{\Delta}_{AB} \;=\; \mathbf{r}_B - \mathbf{r}_A, \qquad r_{AB} \;=\; |\boldsymbol{\Delta}_{AB}|, \qquad f_{AB} \;=\; \frac{k}{r_{AB}^3} \]

The contributions to both bodies' accelerations from the pair \((A, B)\) are then:

(6) \[ \mathbf{a}_A \;\mathrel{+}=\; f_{AB}\,\tilde{m}_B\,\boldsymbol{\Delta}_{AB}, \qquad \mathbf{a}_B \;\mathrel{-}=\; f_{AB}\,\tilde{m}_A\,\boldsymbol{\Delta}_{AB} \]

This double-loop over distinct pairs \(0 \le A < B \le N-1\) is implemented in compute_accelerations() in physics_multiple.py. The loop executes \(N(N-1)/2\) pair evaluations per call — one for each unique pair of bodies — which is the minimum possible for an all-pairs calculation.

Computational cost: Each call to compute_accelerations() costs \(\mathcal{O}(N^2)\) operations. For a fixed timestep, doubling the number of bodies quadruples the CPU time per step. A simulation of 10 bodies therefore takes approximately \(10^2/5^2 = 4\) times longer per step than a simulation of 5 bodies. See the Computational Scaling section for further discussion.

Energy Conservation

The total mechanical energy of the N-body system is the sum of the kinetic energies of all bodies and all pairwise gravitational potential energies:

(7) \[ K \;=\; \sum_{A=0}^{N-1} \tfrac{1}{2}\,m_A^{\mathrm{SI}} \left(v_A^2 + u_A^2 + w_A^2\right) \]
(8) \[ U \;=\; -\sum_{A=0}^{N-1}\sum_{B=A+1}^{N-1} \frac{G\,m_A^{\mathrm{SI}}\,m_B^{\mathrm{SI}}}{r_{AB}} \]
(9) \[ E \;=\; K + U \]

For a gravitationally bound N-body system \(E < 0\). Although plot_multiple.py does not currently plot energy vs. time, \(E\) can be computed from the stored position and velocity arrays returned by run_simulation() and used as a diagnostic for integration quality. A gradual drift in \(E\) indicates accumulated truncation error and can be reduced by tightening eps1 and eps2.

Computational Scaling

The pair-loop in compute_accelerations() is called at every stage of the predictor–corrector, so the total cost scales as:

(10) \[ \text{CPU time} \;\propto\; \text{max\_steps} \times \frac{N(N-1)}{2} \]

This quadratic scaling in \(N\) sets a practical upper limit on the number of bodies for interactive use. The following table gives approximate relative runtimes for equal step counts:

Predictor–Corrector

The integration scheme is a two-stage predictor–corrector applied simultaneously to all \(N\) bodies. Let superscript \(n\) denote the state at time \(t\) and \(n+1\) the state to be determined at \(t + \Delta t\). All position, velocity, and acceleration quantities below are \(N\times3\) arrays — one row per body, three columns for \(x\), \(y\), \(z\).

Stage 1 — Euler Predictor

Predicted positions and velocities are obtained by an explicit Euler step that uses the current acceleration to estimate the second-order correction:

(11) \[ \tilde{\mathbf{r}} \;=\; \mathbf{r}^n + \mathbf{v}^n\,\Delta t + \tfrac{1}{2}\,\mathbf{a}^n\,\Delta t^2 \]
(12) \[ \tilde{\mathbf{v}} \;=\; \mathbf{v}^n + \mathbf{a}^n\,\Delta t \]

The accelerations at the predicted positions, \(\tilde{\mathbf{a}} = \texttt{compute\_accelerations}(\tilde{\mathbf{r}}, \tilde{m})\), are then evaluated.

Stage 2 — Averaged-Acceleration Corrector

Corrected velocities and positions are formed by averaging the start-of-step and predicted accelerations or velocities:

(13) \[ \mathbf{v}^{n+1} \;=\; \mathbf{v}^n + \tfrac{1}{2}\!\left(\mathbf{a}^n + \tilde{\mathbf{a}}\right)\Delta t \]
(14) \[ \mathbf{r}^{n+1} \;=\; \mathbf{r}^n + \tfrac{1}{2}\!\left(\mathbf{v}^n + \mathbf{v}^{n+1}\right)\Delta t \]

These are the same averaged-velocity and averaged-acceleration rules used in Binary and MercPert, now applied to all \(N\) bodies in vectorised form. The corrector is evaluated once per accepted step; there is no inner iteration loop in this implementation.

Adaptive Timestep

Two separate timestep-halving checks are performed at each step. Both compare fractional changes relative to the post-corrector values.

Acceleration Check (eps1)

The maximum component-wise fractional change in acceleration between the Euler predictor and the corrector is computed across all bodies:

(15) \[ \delta_{\mathbf{a}} \;=\; \max_{A,\,i} \frac{\bigl|\tilde{a}_{A,i} - a_{A,i}^{n+1}\bigr|} {\bigl|a_{A,i}^{n+1}\bigr|} \]

If \(\delta_{\mathbf{a}} > \varepsilon_1\), the step is rejected, the working timestep is halved, and the entire predictor–corrector cycle is retried:

(16) \[ \Delta t \;\leftarrow\; \tfrac{1}{2}\,\Delta t \qquad (\delta_{\mathbf{a}} > \varepsilon_1) \]

Position Check (eps2)

The maximum component-wise fractional change in position over the accepted step is also tested:

(17) \[ \delta_{\mathbf{r}} \;=\; \max_{A,\,i} \frac{\bigl|r_{A,i}^{n+1} - r_{A,i}^n\bigr|} {\bigl|r_{A,i}^{n+1}\bigr|} \]

If \(\delta_{\mathbf{r}} > \varepsilon_2\), the timestep is again halved and the step retried. This second check catches situations where positions are changing rapidly even if accelerations appear converged.

Timestep collapse: As in Binary and MercPert, the working timestep can only shrink — it is never grown after an accepted step. Close encounters between bodies force the timestep to very small values and, because the timestep does not recover, can stall the simulation before max_steps is reached. If this occurs, loosen eps1 (at the cost of accuracy) or use the RK45 version of the program (Multiple_RK45), which replaces the predictor–corrector with a modern adaptive integrator that recovers the timestep after a close encounter.

Convergence Criteria

Parameter Symbol Default Role
eps1 \(\varepsilon_1\) \(10^{-2}\) Acceleration-change gate. If the maximum fractional change in any acceleration component between the Euler predictor and the corrector exceeds this value, the step is rejected and \(\Delta t\) is halved.
eps2 \(\varepsilon_2\) \(10^{-3}\) Position-change gate. If the maximum fractional change in any position component over the accepted step exceeds this value, the step is rejected and \(\Delta t\) is halved. Should be smaller than \(\varepsilon_1\).
Design relationship: Setting \(\varepsilon_2 < \varepsilon_1\) ensures that the position check is the tighter constraint, consistent with the defaults \(\varepsilon_1 = 10^{-2}\), \(\varepsilon_2 = 10^{-3}\). Tightening both by a factor of ten typically reduces accumulated energy drift by one to two orders of magnitude at the cost of more step halvings near close encounters.

Output Modes

The output_type parameter selects one of two distinct output and plotting modes:

  • 'trajectories' Positions of all bodies are stored at every accepted integration step. After the run completes, plot_trajectories() renders the full \(xy\)-projection of each body's path as a continuous curve. This mode is ideal for visualising the shape of orbits and interaction sequences, but requires memory proportional to max_steps × n_bodies × 3 floating-point numbers. For 3 bodies and 600,000 steps this is approximately 14 MB — manageable on any modern computer.
  • 'current_positions' A positional snapshot of all bodies is recorded every out_steps accepted steps. After the run completes, plot_current_positions() renders the last snapshot as a scatter plot with one dot per body, labelled with the simulation time. This mode is memory-efficient for long runs and produces a video-like sequence of frames if out_steps is chosen small enough relative to the dynamical timescale.
Choosing between modes: Use 'trajectories' when you want to see the full orbital paths — for example, to identify bound pairs or trace ejection trajectories. Use 'current_positions' when the number of steps is so large that storing all positions would require too much memory, or when you want a movie-like visualisation of the system evolving in real time.

Suggested Scenarios

The following scenarios are drawn from Schutz's original suggestions and are all reproducible by editing parameters in main.py.

Default three-body encounter
Three equal-mass stars (\(1\,M_\odot\) each) interact. Two form a bound binary and the third is ejected with high speed. Try reassigning the masses and observe how the outcome changes.
Binary hardening
Extend the default run to 6,000,000 steps. The ejected star remains bound and returns, colliding again. Each collision transfers energy to the binary (hardening it) while the outer orbit widens.
Out-of-plane velocities
Add a small \(z\)-component of velocity to one or more bodies in the default configuration. Monitor how the three-body interaction changes when the encounter is genuinely three-dimensional rather than planar.
Star cluster (5–10 bodies)
Place 5–10 equal-mass stars at random positions within 0.1 pc of the origin with random velocities below 10 km s−1. Observe whether the cluster expands ("boils away") or remains bound. Then introduce one or two 10 \(M_\odot\) bodies and observe their effect on the lighter stars.
MercPert verification
Configure Multiple with the same masses and initial conditions as MercPert (Sun, Jupiter-mass companion, Mercury test particle with small but non-zero mass). Verify that the trajectory matches MercPert's output. Then increase Mercury's mass to Jupiter-mass and observe the mutual perturbations.
Galaxy collision (toy model)
Use two very massive bodies (\(10^{11}\,M_\odot\)) with initial separation 50 kpc approaching at 100 km s−1, surrounded by smaller bodies representing stars. Observe how the "galaxies" interact and whether a merger results when the smaller bodies carry non-negligible mass.
