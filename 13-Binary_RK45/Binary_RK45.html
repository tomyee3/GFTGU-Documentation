Overview

Binary_RK45 simulates the orbital motion of two gravitationally bound point masses under Newtonian gravity — the same physical problem treated by the companion program Binary, but integrated using the modern, production-quality Runge–Kutta–Fehlberg 4(5) adaptive method provided by scipy.integrate.solve_ivp rather than the hand-coded predictor–corrector of the original Schutz Java program.

Given the initial positions and velocities of both bodies, the program integrates the coupled equations of motion forward in time to t_max, recording the trajectory of each body at every internally chosen step. It then produces an orbital plot in the \(xy\)-plane with equal aspect ratio.

Scope: Binary_RK45 is a purely Newtonian simulation. No relativistic corrections are applied. For post-Newtonian or general-relativistic comparisons see MercPert and RelativisticOrbit in this collection.

Why RK45?

The original Schutz Java program uses a predictor–corrector with a hand-managed adaptive timestep: if the fractional change in acceleration over a step exceeds a threshold \(\varepsilon_1\), the timestep is halved and the step is retried. This strategy works well for nearly circular orbits but becomes fragile for highly eccentric ones. Near periapsis — where the two bodies approach closely and the gravitational acceleration changes rapidly — the required timestep can become extremely small. In practice, the algorithm may halve the timestep so many times that progress through the orbit effectively stalls: even with max_steps set to tens of thousands, only a fraction of one orbit is completed before the step budget is exhausted.

This timestep-collapse problem is not a user error. It is a known limitation of simple halving strategies applied to stiff or rapidly varying problems. The fix is to use a higher-order integrator with a principled, globally controlled error estimate.

RK45 solves this in three ways:

  • Higher-order accuracy. Each accepted step achieves fifth-order accuracy in the local truncation error, compared to the second-order accuracy of the predictor–corrector. More accuracy per step means fewer steps are needed to cover the same arc.
  • Principled step-size selection. scipy chooses each step size so that the estimated local error stays within the user-specified tolerances rtol and atol. Steps grow and shrink smoothly rather than collapsing catastrophically.
  • Dense output. The solver can evaluate the solution at any intermediate time without additional function evaluations, so recording the trajectory at uniformly spaced output times costs almost nothing extra.
Historical note: Schutz's original help file for Binary notes that some choices of initial conditions can cause the program to run very slowly or produce incomplete orbits, and recommends adjusting the timestep and accuracy parameters as a workaround. Binary_RK45 eliminates this class of problem entirely by delegating step-size control to a battle-tested library integrator.

Quick Start

Requirements

Python 3.9 or later with numpy, scipy, and matplotlib:

pip install numpy scipy matplotlib

Running the Program

cd GFTGU-Programs/Binary_RK45
python main.py

The default configuration uses two equal-mass bodies, \(M_A = M_B = 2\times10^{30}\,\mathrm{kg}\) (approximately one solar mass each), placed symmetrically at \(x_A = -5\times10^{10}\,\mathrm{m}\) and \(x_B = +5\times10^{10}\,\mathrm{m}\) on the \(x\)-axis, with equal and opposite transverse velocities of \(\pm\,1\times10^{4}\,\mathrm{m\,s^{-1}}\). The integration runs for \(t_{\max} = 10^9\,\mathrm{s}\) (approximately 31.7 years).

Adjusting the Simulation

All initial conditions and run parameters are set near the top of main.py. The key quantities to vary are:

  • MA, MB — masses of the two bodies (kg)
  • xInitA, xInitB, yInitA, yInitB — starting positions (m)
  • uInitA, uInitB — transverse velocity components \(\dot y\) (m s−1)
  • vInitA, vInitB — radial velocity components \(\dot x\) (m s−1)
  • t_max — total integration time (s)
  • rtol, atol — relative and absolute error tolerances passed to solve_ivp

No other file needs editing for typical use.

Choosing Output

The current version of plot_binary.py renders orbital trajectories in the \(xy\)-plane. Additional output types matching the original Java program (velocity space, position vs. time, velocity vs. time, energy vs. time) are straightforward to add by passing the solver output arrays to additional matplotlib calls — see Output Types.

Program Files

main.py
Entry Point & Configuration

Defines every initial condition and run parameter — masses, starting positions and velocities, the simulation end time t_max, and the solver tolerances rtol and atol. Calls run_binary_orbit() to integrate the trajectory and plot_orbits() to display the result. All user-adjustable settings live here; no other file needs editing for typical use.

physics_binary.py
Physics Engine

Defines the physical constant \(G = 6.6726\times10^{-11}\, \mathrm{N\,m^2\,kg^{-2}}\) and the right-hand-side function acceleration_binary(t, state, MA, MB). The function unpacks the eight-component state vector \((x_A, y_A, x_B, y_B, v_A, u_A, v_B, u_B)\), computes the displacement \(\Delta x = x_A - x_B\), \(\Delta y = y_A - y_B\) and scalar separation \(r_{AB}\), evaluates the Newtonian acceleration on each body, and returns the full state derivative as a NumPy array \((v_A,\, u_A,\, v_B,\, u_B,\, a_{x,A},\, a_{y,A},\, a_{x,B},\, a_{y,B})\). This function is the sole interface between the physics and the integrator.

driver_binary.py
Integration Driver

Provides run_binary_orbit(MA, MB, y0, t_max, rtol, atol). Assembles the initial state vector \(\mathbf{y}_0\) from the individual position and velocity components, then calls scipy.integrate.solve_ivp with method='RK45', passing acceleration_binary as the right-hand side. The solver returns a scipy.integrate.OdeSolution object whose .t and .y arrays contain the accepted time points and state vectors respectively. The driver unpacks these into named arrays (xA, yA, xB, yB, etc.) and returns them as a plain Python dictionary for use by plot_binary.py.

plot_binary.py
Visualisation

Provides plot_orbits(data), which accepts the dictionary returned by run_binary_orbit() and renders an 8 × 8 inch matplotlib figure showing the trajectories of both bodies in the \(xy\)-plane with equal aspect ratio (plt.axis("equal")), axis labels in metres, a legend identifying Body A and Body B, a grid, and the title Binary Orbits (RK45).

Two-Body Problem

Two point masses \(M_A\) and \(M_B\) interact solely through Newtonian gravitational attraction. Newton's law of universal gravitation gives the magnitude of the mutual force:

(1) \[ F \;=\; \frac{G\,M_A\,M_B}{r^2} \]

where \(G = 6.6726\times10^{-11}\,\mathrm{N\,m^2\,kg^{-2}}\) is Newton's gravitational constant and

(2) \[ r \;=\; \sqrt{(x_A - x_B)^2 + (y_A - y_B)^2} \]

is the scalar separation between the two bodies. The motion is confined to the \(xy\)-plane. Following the convention of the source code and the original Schutz Java program, \(v\) denotes the \(x\)-component of velocity (\(\dot x\)) and \(u\) denotes the \(y\)-component (\(\dot y\)).

Coordinate convention: Each body's state is stored as \((x,\,y,\,v,\,u)\) where \(v = \dot x\) and \(u = \dot y\), following the original Schutz Java convention retained in all Python ports.

With equal masses and symmetric initial conditions the centre of mass remains at the origin throughout the simulation. Varying the mass ratio or the initial transverse speed produces elliptical orbits of arbitrary eccentricity.

Equations of Motion

Applying Newton's second law component by component, with the force directed along the line joining the two bodies, yields four coupled second-order ODEs. Define the displacement of A relative to B: \(\Delta x = x_A - x_B\), \(\Delta y = y_A - y_B\), and \(r^3 = \bigl((\Delta x)^2 + (\Delta y)^2\bigr)^{3/2}\).

Body A

(3) \[ \ddot x_A \;=\; -\frac{G\,M_B\,\Delta x}{r^3}, \qquad \ddot y_A \;=\; -\frac{G\,M_B\,\Delta y}{r^3} \]

Body B

(4) \[ \ddot x_B \;=\; +\frac{G\,M_A\,\Delta x}{r^3}, \qquad \ddot y_B \;=\; +\frac{G\,M_A\,\Delta y}{r^3} \]

The sign flip between bodies reflects Newton's third law: the force on B is equal and opposite to the force on A. These four second-order ODEs are rewritten as the eight first-order ODEs that form the state derivative returned by acceleration_binary().

The complete system state at any instant is the eight-component vector \(\mathbf{y} = (x_A,\,y_A,\,x_B,\,y_B,\,v_A,\,u_A,\,v_B,\,u_B)\). Its time derivative is

(5) \[ \dot{\mathbf{y}} \;=\; \bigl(v_A,\;u_A,\;v_B,\;u_B,\; a_{x,A},\;a_{y,A},\;a_{x,B},\;a_{y,B}\bigr) \]

where \(a_{x,A}\), \(a_{y,A}\), \(a_{x,B}\), \(a_{y,B}\) are given by equations (3) and (4). This is the vector passed to solve_ivp at every internal evaluation point.

Energy Conservation

The total mechanical energy is the sum of the kinetic energies of both bodies and their mutual gravitational potential energy.

Kinetic Energy

(6) \[ K \;=\; \tfrac{1}{2}\,M_A\!\left(v_A^2 + u_A^2\right) + \tfrac{1}{2}\,M_B\!\left(v_B^2 + u_B^2\right) \]

Gravitational Potential Energy

(7) \[ U \;=\; -\frac{G\,M_A\,M_B}{r} \]

Total Mechanical Energy

(8) \[ E \;=\; K + U \]

For a gravitationally bound orbit \(E < 0\) and is a conserved quantity. The RK45 integrator preserves energy far better than the predictor–corrector for the same amount of computational work, because its error estimate is based on a genuine fifth-order local truncation error bound rather than an empirical fractional-change heuristic.

Monitoring energy drift: Although plot_binary.py currently renders only orbital trajectories, energy conservation can be checked easily by computing \(E(t)\) from the solver output arrays vA, uA, vB, uB, xA, yA, xB, yB returned by run_binary_orbit() and plotting it vs. t. A flat line confirms that the tolerances are tight enough. If drift is visible, halve rtol and atol.

Orbit Closure

Unlike the original Binary predictor–corrector, Binary_RK45 does not monitor orbit closure automatically. Integration simply runs from \(t = 0\) to \(t = t_{\max}\) and the solver records the trajectory at every internally accepted step. The user is responsible for choosing a t_max that spans the desired number of orbits.

For the default equal-mass configuration, the orbital period can be estimated from Kepler's third law applied to the relative orbit of separation \(d = r_{AB,0}\):

(9) \[ T \;=\; 2\pi \sqrt{\frac{d^3}{G\,(M_A + M_B)}} \]

Setting t_max to a small integer multiple of \(T\) ensures the plot shows complete, closed orbits for a bound system.

Unbound orbits: If the initial velocities exceed the escape velocity the bodies fly apart and the orbit never closes. You can detect this by checking that the total energy \(E\) (equation 8) is negative. If \(E \ge 0\), increase t_max and you will simply see the bodies receding without returning — a correct result, not a numerical failure.

RK45 Method

The Runge–Kutta–Fehlberg 4(5) method — commonly called RK45 — is an embedded pair of explicit Runge–Kutta schemes of order 4 and order 5 that share the same function evaluations. At each step of width \(h\), the method evaluates the right-hand side \(\mathbf{f}(t,\mathbf{y})\) at six (or seven, in the Dormand–Prince variant used by SciPy) internal stages \(k_1, \ldots, k_s\).

Stage Evaluations

The internal stages are defined by

(10) \[ \mathbf{k}_i \;=\; \mathbf{f}\!\left( t_n + c_i\,h,\; \mathbf{y}_n + h\sum_{j=1}^{i-1} a_{ij}\,\mathbf{k}_j \right), \qquad i = 1, \ldots, s \]

where the coefficients \(c_i\) and \(a_{ij}\) are fixed by the Butcher tableau of the chosen method (Dormand–Prince in SciPy's default RK45 implementation).

Fourth- and Fifth-Order Solutions

Two solution approximations are formed as weighted combinations of the stage values:

(11) \[ \mathbf{y}_{n+1}^{(4)} \;=\; \mathbf{y}_n + h\sum_{i=1}^{s} b_i^*\,\mathbf{k}_i \qquad \text{(4th order)} \]
(12) \[ \mathbf{y}_{n+1}^{(5)} \;=\; \mathbf{y}_n + h\sum_{i=1}^{s} b_i\,\mathbf{k}_i \qquad \text{(5th order, used to advance)} \]

The fifth-order solution is accepted as the new state. The fourth-order solution is used only to estimate the local error.

Local Error Estimate

The difference between the two solutions provides a per-component estimate of the local truncation error:

(13) \[ \mathbf{e}_{n+1} \;=\; \mathbf{y}_{n+1}^{(5)} - \mathbf{y}_{n+1}^{(4)} \;=\; h\sum_{i=1}^{s} (b_i - b_i^*)\,\mathbf{k}_i \]

This estimate is free — it uses stage values already computed for the accepted step.

State Vector

The eight-component state vector passed to solve_ivp is assembled in driver_binary.py as:

(14) \[ \mathbf{y}_0 \;=\; \begin{pmatrix} x_A^{(0)} \\ y_A^{(0)} \\ x_B^{(0)} \\ y_B^{(0)} \\ v_A^{(0)} \\ u_A^{(0)} \\ v_B^{(0)} \\ u_B^{(0)} \end{pmatrix} \]

At each internal evaluation point solve_ivp calls acceleration_binary(t, y, MA, MB), which returns \(\dot{\mathbf{y}}\) as in equation (5). The solver never calls any other function; all physics is encapsulated in physics_binary.py.

After integration, the solution arrays are extracted from the sol.y attribute of the returned OdeResult object by index: 

xA = sol.y[0],  yA = sol.y[1],  xB = sol.y[2],  yB = sol.y[3]
vA = sol.y[4],  uA = sol.y[5],  vB = sol.y[6],  uB = sol.y[7]

Error Control

scipy's RK45 implementation controls step size so that the estimated local error satisfies a mixed absolute-relative tolerance criterion for every component of the state vector. For component \(i\):

(15) \[ |e_i| \;\le\; \mathrm{atol} + \mathrm{rtol} \cdot |y_i| \]

If this condition is satisfied for all eight components the step is accepted and the solver advances. Otherwise the step is rejected and retried with a smaller \(h\), chosen by the standard PI controller. After each accepted step the solver proposes the next step size as

(16) \[ h_{\mathrm{new}} \;=\; h \cdot \min\!\left( \text{factor}_{\max},\; \max\!\left(\text{factor}_{\min},\; \text{safety} \cdot \left(\frac{\text{tol}}{\|e\|}\right)^{1/5} \right) \right) \]

where safety = 0.9, factor_min = 0.2, and factor_max = 10 are SciPy defaults. This produces smooth, efficient step-size variation that adapts to the local dynamics without catastrophic collapse near periapsis.

Parameter Default Role
rtol \(10^{-9}\) Relative tolerance — controls error proportional to the magnitude of each state component. Dominant for large coordinates (far from origin).
atol \(10^{-9}\) Absolute tolerance — controls error in absolute terms. Dominant when a state component is near zero (e.g. \(y\)-velocity at apoapsis for a symmetric orbit).
Tightening tolerances: The defaults \(\mathrm{rtol} = \mathrm{atol} = 10^{-9}\) are tight enough for excellent energy conservation over many orbits. Loosening them (to \(10^{-6}\), say) speeds up the computation at the cost of visible energy drift. Tightening them further (to \(10^{-12}\)) may be useful for very long runs or for close-passage orbits where rounding error becomes relevant, though at significantly increased compute time.

Output Types

The current release of plot_binary.py renders orbital trajectories. The following additional output modes — all present in the original Schutz Java program — are straightforward to implement from the arrays returned by run_binary_orbit():

  • "orbits" Orbital trajectories of both bodies in the \(xy\)-plane with equal aspect ratio. Currently implemented.
  • "velocity space" Velocity-space trajectories — \(u\) vs. \(v\) for each body. For circular orbits these traces are circles; elliptical orbits produce ellipses.
  • "position vs. time body A" \(x_A(t)\) and \(y_A(t)\) as functions of time for body A.
  • "position vs. time body B" \(x_B(t)\) and \(y_B(t)\) as functions of time for body B.
  • "velocity vs. time body A" \(v_A(t)\) and \(u_A(t)\) as functions of time for body A.
  • "velocity vs. time body B" \(v_B(t)\) and \(u_B(t)\) as functions of time for body B.
  • "energy vs time" Kinetic \(K(t)\), gravitational potential \(U(t)\), and total mechanical energy \(E(t)\) vs. time. Primary diagnostic for assessing numerical accuracy.

Comparison with Binary

Binary_RK45 and Binary solve the same physical problem with different numerical strategies. The table below summarises the key differences.

Feature Binary (predictor–corrector) Binary_RK45 (scipy RK45)
Integrator order 2nd order (averaged acceleration) 5th order (Dormand–Prince)
Step-size control Halving only; no recovery growth beyond 10%/step PI controller; smooth growth and shrink
Eccentric orbit robustness Timestep collapse possible near periapsis Robust; adapts smoothly near periapsis
Energy conservation Good for circular/mildly elliptical orbits Excellent; \(10^{-9}\) tolerances suppress drift
Dependencies matplotlib only numpy, scipy, matplotlib
Orbit-closure detection Automatic (angular tracking) None — user sets t_max
User-tunable accuracy eps1, eps2, dt rtol, atol
Output types 7 (orbits, vel-space, pos/vel/energy vs. time) 1 currently (orbits); others easy to add
Which to use? For nearly circular orbits and quick experimentation, Binary's predictor–corrector is self-contained and requires only matplotlib. For eccentric orbits, long runs, or whenever energy conservation matters quantitatively, Binary_RK45 is the better choice — it will complete the orbit where Binary stalls, and its energy error is orders of magnitude smaller.
