Overview

Multiple_RK45 simulates the gravitational motion of any number of point masses interacting under Newtonian gravity in full three dimensions — the same physical problem treated by the companion program Multiple, but integrated using the modern, production-quality Runge–Kutta–Fehlberg 4(5) adaptive method provided by scipy.integrate.solve_ivp rather than the hand-coded predictor–corrector of the original Schutz Java program.

Given the masses, initial positions, and initial velocities of all bodies, the program integrates the coupled equations of motion forward in time to t_max, recording all bodies' positions at uniform output intervals dt_output. It then produces either a trajectory plot of all bodies projected onto the \(xy\)-plane or a scatter plot of current positions at each output instant.

Scope: Multiple_RK45 uses Newtonian gravity only. No relativistic corrections are applied. The simulation is three-dimensional; all three position and velocity components are integrated, but visualisation is projected onto the \(xy\)-plane.

Why RK45?

The original Schutz Java program uses a predictor–corrector with a hand-managed adaptive timestep: when the fractional change in any body's acceleration over a step exceeds the threshold \(\varepsilon_1\), the timestep is halved and the step is retried. Because the timestep is never subsequently lengthened, the simulation can stall — even with max_steps set to hundreds of thousands, only a fraction of the intended simulation time may be covered before the step budget is exhausted.

In the N-body setting this problem is more severe than in the two-body case: there are \(N(N-1)/2\) pairs, and a single close encounter between any two bodies can collapse the timestep for all bodies, stalling the entire simulation. Schutz's original help file acknowledges this, noting that certain initial conditions cause the program to run very slowly and recommending parameter adjustments as a workaround — at the cost of accuracy.

RK45 eliminates this problem in three ways:

  • Higher-order accuracy. Each accepted step achieves fifth-order accuracy in the local truncation error — far better than the second-order predictor–corrector — so fewer internal steps are needed to cover the same arc of the trajectory.
  • Principled step-size selection. SciPy chooses each internal step size to keep the estimated local error within rtol and atol. Steps shrink near close encounters and grow again in quiet phases — smoothly and automatically, without ever collapsing to zero.
  • Output independent of integration steps. The solver evaluates the solution at the uniform output grid t_eval (spaced by dt_output) using dense output interpolation. Requesting more output points never forces additional internal steps.
Historical note: Schutz's original Multiple help file warns that close encounters cause the Java program to run very slowly and asks users to increase the accuracy threshold \(\varepsilon_1\) as a workaround. Multiple_RK45 eliminates this class of problem: any physically valid set of initial conditions will integrate completely to t_max, with internal step sizes chosen automatically to maintain the requested accuracy throughout.

Quick Start

Requirements

Python 3.9 or later with numpy, scipy, and matplotlib:

pip install numpy scipy matplotlib

Running the Program

cd GFTGU-Programs/Multiple_RK45
python main.py

The default configuration simulates three equal-mass bodies (\(M \approx M_\odot\) each) in a three-body encounter designed to reproduce the Schutz textbook example: two bodies form a gravitationally bound pair while the third is ejected. The integration runs to t_max with positions recorded every dt_output seconds.

Adjusting the Simulation

All parameters are set near the top of main.py:

  • n_bodies — number of bodies
  • masses — list of masses in kg, one per body
  • positions — list of [x, y, z] initial positions in metres
  • velocities — list of [vx, vy, vz] initial velocities in m s−1
  • t_max — total simulation time in seconds
  • dt_output — interval between recorded output points in seconds; does not affect integrator accuracy
  • rtol, atol — relative and absolute error tolerances passed to solve_ivp
  • output_type — 'trajectories' or 'positions'

Make sure masses, positions, and velocities each contain exactly n_bodies entries. No other file needs editing for typical use.

Memory usage: In 'trajectories' mode, positions are stored at every output point. The total storage is proportional to (t_max / dt_output) × n_bodies × 3 floating-point numbers. For large runs with many bodies, increase dt_output or switch to 'positions' mode to keep memory use manageable.

Program Files

main.py
Entry Point & Configuration

Defines all simulation parameters — number of bodies, masses (in kg), initial positions and velocities, the end time t_max, the output sampling interval dt_output, the solver tolerances rtol and atol, and the output mode. Calls run_simulation() to integrate the trajectory, then dispatches to plot_trajectories() or plot_positions() depending on output_type. All user-adjustable settings live here; no other file needs editing for typical use.

physics_multiple.py
Physics Engine

Defines \(G = 6.6726\times10^{-11}\,\mathrm{N\,m^2\,kg^{-2}}\) and the right-hand-side function accelerations(t, y, masses). This function unpacks the \(6N\)-component state vector into \(N\times3\) position and velocity arrays, iterates over all \(N(N-1)/2\) distinct body pairs using a double loop, applies Newton's third law to compute both bodies' contributions from a single pair calculation, and returns the full \(6N\)-component state derivative \((\dot x, \dot y, \dot z, a_x, a_y, a_z)\) for every body as a flat NumPy array. This function is the sole interface between the physics and the integrator.

driver_multiple.py
Integration Driver

Provides run_simulation(n_bodies, masses, positions, velocities, t_max, dt_output, rtol, atol). The driver assembles the \(6N\)-component initial state vector \(\mathbf{y}_0\) from the user-supplied position and velocity lists, builds the output time grid t_eval = np.arange(0, t_max, dt_output), and calls scipy.integrate.solve_ivp with method='RK45', passing accelerations as the right-hand side. After the solver returns, it unpacks sol.y into a list of per-body trajectory arrays of shape (len(t_eval), 3) — one array per body — and returns (sol.t, trajectories).

plot_multiple.py
Visualisation

Provides two functions, both accepting the list of per-body trajectory arrays and a list of label strings returned by run_simulation(). plot_trajectories(trajectories, labels) renders the full \(xy\)-projection of every body's trajectory in a single matplotlib figure, with bodies coloured in sequence (red, green, blue, orange, purple, brown, cycling for \(N > 6\)) and initial positions marked. plot_positions(trajectories, labels) renders a scatter plot of all bodies' final positions. Both functions use equal aspect ratio and display a legend.

N-Body Problem

The gravitational N-body problem describes the motion of \(N\) point masses \(m_0, m_1, \ldots, m_{N-1}\) (in kg) interacting pairwise through Newtonian gravity. Every body is attracted toward every other; there are no central forces, no prescribed orbits, and no test-particle approximation — all bodies are integrated on equal footing.

The motion is fully three-dimensional. Each body's state at any instant is described by its position vector \(\mathbf{r}_A = (x_A, y_A, z_A)\) and velocity vector \(\dot{\mathbf{r}}_A = (v_A, u_A, w_A)\), where \(v\), \(u\), \(w\) denote the \(x\)-, \(y\)-, and \(z\)-components of velocity. The complete system state is the \(6N\)-component vector formed by concatenating all bodies' positions and velocities.

Projection: The simulation is three-dimensional but plot_multiple.py projects trajectories onto the \(xy\)-plane. To view a different projection, rotate the initial conditions so that the plane of interest coincides with \(xy\), or post-process the stored trajectory arrays.

Equations of Motion

The gravitational acceleration on body \(A\) due to body \(B\) is directed from \(A\) toward \(B\) with magnitude \(Gm_B/r_{AB}^2\):

(1) \[ \mathbf{a}_{A \leftarrow B} \;=\; \frac{G\,m_B}{r_{AB}^3}\,\bigl(\mathbf{r}_B - \mathbf{r}_A\bigr) \]

where \(r_{AB} = |\mathbf{r}_B - \mathbf{r}_A|\) is the scalar separation. The total acceleration on body \(A\) is the vector sum over all other bodies:

(2) \[ \ddot{\mathbf{r}}_A \;=\; \sum_{\substack{B=0 \\ B \neq A}}^{N-1} \frac{G\,m_B}{r_{AB}^3}\,\bigl(\mathbf{r}_B - \mathbf{r}_A\bigr) \]

These \(N\) vector equations (each with three components) constitute the \(3N\) coupled second-order ODEs of the N-body problem. Rewritten as \(6N\) first-order ODEs by introducing the velocities as auxiliary variables, the full system state derivative is:

(3) \[ \dot{\mathbf{y}} \;=\; \bigl( \dot{\mathbf{r}}_0,\; \dot{\mathbf{r}}_1,\;\ldots,\; \dot{\mathbf{r}}_{N-1},\; \ddot{\mathbf{r}}_0,\; \ddot{\mathbf{r}}_1,\;\ldots,\; \ddot{\mathbf{r}}_{N-1} \bigr) \]

This is the vector returned by accelerations(t, y, masses) in physics_multiple.py and evaluated at every internal stage of the RK45 step.

Pair-Loop Acceleration

The key efficiency of compute_accelerations() is that the mutual contributions of any pair \((A, B)\) are computed together in a single pass, exploiting Newton's third law:

(4) \[ \mathbf{a}_{A \leftarrow B} \;=\; -\,\mathbf{a}_{B \leftarrow A} \]

For each pair \(0 \le A < B \le N-1\), define the displacement vector and common geometric factor:

(5) \[ \boldsymbol{\Delta}_{AB} \;=\; \mathbf{r}_B - \mathbf{r}_A, \qquad r_{AB} \;=\; |\boldsymbol{\Delta}_{AB}|, \qquad f_{AB} \;=\; \frac{G}{r_{AB}^3} \]

Both bodies' accelerations are then updated in a single step:

(6) \[ \mathbf{a}_A \;\mathrel{+}=\; f_{AB}\,m_B\,\boldsymbol{\Delta}_{AB}, \qquad \mathbf{a}_B \;\mathrel{-}=\; f_{AB}\,m_A\,\boldsymbol{\Delta}_{AB} \]

The loop runs over all \(N(N-1)/2\) distinct pairs — the minimum necessary for an all-pairs calculation. compute_accelerations() is called at every internal stage evaluation of the RK45 step, so the total number of pair evaluations per accepted step is \(6 \times N(N-1)/2\) (six stages per Dormand–Prince RK45 step, with FSAL reusing the last stage of one step as the first of the next).

Energy Conservation

The total mechanical energy of the system is conserved exactly in Newtonian gravity and serves as the primary diagnostic for numerical accuracy:

(7) \[ K \;=\; \sum_{A=0}^{N-1} \tfrac{1}{2}\,m_A \left(v_A^2 + u_A^2 + w_A^2\right) \]
(8) \[ U \;=\; -\sum_{A=0}^{N-1}\sum_{B=A+1}^{N-1} \frac{G\,m_A\,m_B}{r_{AB}} \]
(9) \[ E \;=\; K + U \;=\; \text{constant} \]

Although plot_multiple.py does not currently plot energy vs. time, \(E(t)\) can be computed from the trajectory arrays returned by run_simulation()rtol and atol are tight enough. If drift is visible, halve both tolerances.

Tip: For a gravitationally bound N-body system \(E < 0\). If \(E \ge 0\) at the initial conditions, the system is unbound and the bodies will disperse to infinity — a physically correct result, not a numerical failure.

Computational Scaling

Each internal RK45 stage calls accelerations() once, evaluating \(N(N-1)/2\) pairs. With 6 stages per accepted step (5 unique plus FSAL), the cost per accepted step scales as:

(10) \[ \text{cost per step} \;\propto\; 6 \times \frac{N(N-1)}{2} \;=\; 3\,N(N-1) \]

The quadratic scaling in \(N\) sets a practical upper limit for interactive use:

Professional codes use hierarchical tree algorithms to reduce the scaling from \(\mathcal{O}(N^2)\) to \(\mathcal{O}(N \log N)\), enabling simulations of millions of particles. Multiple_RK45 is designed for educational exploration with up to roughly a dozen bodies, where the all-pairs approach is exact and its cost is manageable.

RK45 Method

The Runge–Kutta–Fehlberg 4(5) method — RK45 — is an embedded pair of explicit Runge–Kutta schemes of orders 4 and 5 that share the same function evaluations. SciPy's implementation uses the Dormand–Prince tableau. At each step of width \(h\), six stage evaluations \(\mathbf{k}_1, \ldots, \mathbf{k}_6\) are computed — with the seventh stage of the next step free via the FSAL (First Same As Last) property.

Stage Evaluations

(11) \[ \mathbf{k}_i \;=\; \mathbf{f}\!\left( t_n + c_i\,h,\; \mathbf{y}_n + h\sum_{j=1}^{i-1} a_{ij}\,\mathbf{k}_j \right), \quad i = 1, \ldots, 6 \]

where \(\mathbf{f}\) is accelerations and the coefficients \(c_i\), \(a_{ij}\) are fixed by the Dormand–Prince Butcher tableau. Because the N-body ODE is autonomous (no explicit time dependence), the \(c_i\) factors do not change the physics — but they are retained for generality.

Fifth-Order Solution (Accepted State)

(12) \[ \mathbf{y}_{n+1} \;=\; \mathbf{y}_n + h\sum_{i=1}^{6} b_i\,\mathbf{k}_i \qquad \text{(5th order — used to advance)} \]

Fourth-Order Companion (Error Estimate Only)

(13) \[ \hat{\mathbf{y}}_{n+1} \;=\; \mathbf{y}_n + h\sum_{i=1}^{6} b_i^*\,\mathbf{k}_i \qquad \text{(4th order — used only for error estimate)} \]

Local Error Estimate

(14) \[ \mathbf{e}_{n+1} \;=\; \mathbf{y}_{n+1} - \hat{\mathbf{y}}_{n+1} \;=\; h\sum_{i=1}^{6}(b_i - b_i^*)\,\mathbf{k}_i \]

This estimate is free — it uses stage values already computed for the accepted step. It drives the step-size controller described in Error Control & Output Sampling.

State Vector

The \(6N\)-component state vector assembled in driver_multiple.py is ordered as positions first, velocities second:

(15) \[ \mathbf{y}_0 \;=\; \bigl( x_0^{(0)},\, y_0^{(0)},\, z_0^{(0)},\; x_1^{(0)},\, y_1^{(0)},\, z_1^{(0)},\; \ldots,\; v_0^{(0)},\, u_0^{(0)},\, w_0^{(0)},\; v_1^{(0)},\, u_1^{(0)},\, w_1^{(0)},\; \ldots \bigr) \]

That is, the first \(3N\) components are all bodies' \((x,y,z)\) positions; the second \(3N\) components are all bodies' \((v_x,v_y,v_z)\) velocities. After integration, driver_multiple.py unpacks sol.y into a list of \(N\) per-body trajectory arrays, each of shape (len(t_eval), 3): 

times        = sol.t                            # shape (len(t_eval),)
trajectories = [sol.y[3*i : 3*(i+1), :].T      # positions of body i
                for i in range(n_bodies)]        # shape (len(t_eval), 3) each

These arrays are passed directly to plot_trajectories() or plot_positions().

Error Control & Output Sampling

Tolerance Criterion

SciPy's RK45 implementation accepts a step only when the estimated local error satisfies a mixed absolute-relative criterion for every component of the \(6N\)-dimensional state vector. For component \(i\):

(16) \[ |e_i| \;\le\; \mathrm{atol} + \mathrm{rtol} \cdot |y_i| \]

If any component violates this bound the step is rejected and retried at a smaller \(h\). After each accepted step the solver proposes the next step size using the standard PI controller:

(17) \[ h_{\mathrm{new}} \;=\; h \cdot \min\!\left( 10,\;\max\!\left(0.2,\; 0.9 \cdot \left(\frac{\mathrm{tol}}{\|\mathbf{e}\|}\right)^{1/5} \right) \right) \]

This produces smooth, efficient step-size variation that contracts near close encounters and expands in quiet orbital phases — automatically and without user intervention.

Output Sampling

The driver builds t_eval = np.arange(0, t_max, dt_output) and passes it to solve_ivp. The solver uses dense-output interpolation to evaluate the solution at every point in t_eval without requiring additional right-hand-side evaluations. This means:

  • Requesting finer output (smaller dt_output) does not slow the integration.
  • Output points are always uniformly spaced in time, regardless of where the internal adaptive steps fall.
  • The internal steps near close encounters may be much shorter than dt_output, and this is handled transparently.

Variable Default Unit Description
n_bodies 3 — Number of bodies. Must match lengths of masses, positions, velocities.

positions see main.py m List of \(N\) initial position vectors [x, y, z].

Output Modes

Set output_type in main.py to one of the following values:

Value Function called What is displayed Best for
'trajectories' plot_trajectories() Full \(xy\)-projected path of every body from \(t=0\) to \(t_{\max}\) Seeing the complete orbital history; identifying bound pairs and ejections
'positions' plot_positions() Scatter of all bodies' positions at the final output time Large \(N\) or long runs where the full trajectory would be unreadable

Suggested Scenarios

These starting points are drawn from Schutz's original Multiple examples. All are stable starting points for Multiple_RK45; none will stall due to timestep collapse.

Default Three-Body Encounter
Keep n_bodies = 3, equal masses, the default positions and velocities. Two stars form a bound binary; the third is ejected. A clean demonstration of the most common outcome of stellar three-body encounters.
Figure-Eight Orbit
Three equal masses with the specific initial conditions of the Chenciner–Montgomery figure-eight solution: all three bodies chase each other around a figure-eight curve indefinitely. A remarkable example of a stable periodic orbit in the three-body problem.
Binary + Intruder
Set two bodies as a tight bound pair and a third as a distant intruder approaching with just above escape velocity. Depending on the impact parameter, the intruder may be captured, exchange partners with one of the binary, or be deflected. Increase t_max to follow the outcome to completion.
Out-of-Plane Motion
Give one body a non-zero \(z\)-velocity component. The orbit precesses out of the \(xy\)-plane; the trajectory plot shows the projected path becoming more complex. Demonstrates why full 3D integration is necessary even for near-planar systems.
Hierarchical Triple
Set n_bodies = 3 with a close inner binary (bodies 0 and 1) and a wide outer body (body 2) orbiting their centre of mass at 5–10× the inner separation. A hierarchical triple is long-term stable; increase t_max to verify that the inner binary remains intact over many outer orbital periods.
Small Star Cluster
Set n_bodies = 5 or 6 with random positions within a sphere and velocities drawn from a Maxwellian distribution. Switch to output_type = 'positions' and use a large t_max to watch the cluster evolve: core collapse, binary hardening, and eventual evaporation of low-mass members are all observable.
