Overview

MercPert_RK45 simulates the trajectory of a Mercury-like test body orbiting under the combined gravitational influence of a central binary system — a Sun-like star and a massive planetary companion in mutual circular orbit. This is the same restricted three-body problem treated by the companion program MercPert, but integrated using the modern, production-quality Runge–Kutta–Fehlberg 4(5) adaptive method provided by scipy.integrate.solve_ivp, rather than the hand-coded predictor–corrector of the original Schutz Java program.

The two binary members move on analytically prescribed circular orbits while Mercury's trajectory is integrated numerically. The program advances Mercury's state from \(t = 0\) to t_max, recording all three bodies' positions at uniform output intervals dt_output, and produces an orbital plot when finished.

Scope: MercPert_RK45 is a purely Newtonian simulation. No post-Newtonian or general-relativistic corrections are applied. Mercury is treated as a true test particle — its mass is negligible and it exerts no gravitational force on either binary member.

Why RK45?

The original Schutz Java program uses a predictor–corrector with a hand-managed adaptive timestep. When the fractional change in Mercury's acceleration over a step exceeds the tolerance \(\varepsilon_1\), the timestep is halved and the step is retried. Because the timestep is never subsequently lengthened, the simulation can stall: even with max_steps set to tens of thousands, only a small fraction of a full orbit may be computed before the step budget is exhausted.

This timestep-collapse problem is most acute near close approaches of Mercury to either binary member — precisely the events that make the simulation physically interesting. Schutz's original help file acknowledges this directly, noting that close Jupiter approaches force repeated halving and recommending that users loosen \(\varepsilon_1\) as a workaround, at the cost of reduced accuracy. MercPert_RK45 eliminates this problem entirely by delegating timestep control to a battle-tested library integrator.

RK45 solves the stalling problem in three ways:

  • Higher-order accuracy. Each accepted step achieves fifth-order accuracy in the local truncation error — far better than the second-order predictor–corrector — so fewer steps are needed to cover the same arc.
  • Principled step-size selection. SciPy chooses each internal step size to keep the estimated local error within the user-specified tolerances rtol and atol. Steps shrink near close approaches and grow again in quiet phases automatically — without ever collapsing to zero.
  • Output independent of integration steps. The solver evaluates the solution at the uniform output grid t_eval (spaced by dt_output) using dense output interpolation. Recording more output points never forces additional internal steps.
Historical note: Schutz's original help file for MercPert warns that certain parameter combinations can cause the Java program to run very slowly or produce an incomplete trajectory, and asks users to adjust accuracy thresholds accordingly. MercPert_RK45 eliminates this class of problem: any combination of masses and initial conditions that produces a physically valid trajectory will integrate completely to t_max.

Quick Start

Requirements

Python 3.9 or later with numpy, scipy, and matplotlib:

pip install numpy scipy matplotlib

Running the Program

cd GFTGU-Programs/MercPert_RK45
python main.py

The default configuration places a Sun-like star (\(M_\odot = 1.98847\times10^{30}\,\mathrm{kg}\)) and a super-Jupiter companion (0.1 solar masses, roughly 100 Jupiter masses) in a circular mutual orbit at 0.7 AU separation. Mercury starts at 0.5 AU from the barycentre on the positive \(x\)-axis with a purely transverse velocity of \(30{,}000\,\mathrm{m\,s^{-1}}\). The integration runs to \(t_{\max} = 10^8\,\mathrm{s}\) (approximately 3.2 years), sampling the trajectory every 2,000 s.

Adjusting the Simulation

All parameters are set near the top of main.py:

  • m_sun_solar, m_planet_solar — masses in solar-mass units
  • binary_separation — binary centre-to-centre distance (m)
  • x_init_merc, y_init_merc — Mercury's initial position (m)
  • vx_init_merc, vy_init_merc — Mercury's initial velocity (m s−1)
  • t_max — total simulation time (s)
  • dt_output — output sampling interval (s); does not affect integrator accuracy
  • rtol, atol — relative and absolute error tolerances

No other file needs editing for typical use.

Covering more orbits: To see more orbital cycles simply increase t_max. Unlike the Java program, you never need to increase dt_output or loosen tolerances just to complete a run — RK45 adapts its internal step sizes automatically regardless of how long the simulation runs.

Program Files

main.py
Entry Point & Configuration

Defines all initial conditions and run parameters — binary masses, separation, Mercury's starting position and velocity, the simulation end time t_max, the output sampling interval dt_output, and the solver tolerances rtol and atol. Calls run_mercpert() to integrate the trajectory and plot_orbits() to display the result. All user-adjustable settings live here; no other file needs editing for typical use.

physics_mercpert.py
Physics Engine

Defines the physical constants (\(G\), \(M_\odot\), \(\mathrm{AU}\)) and three functions. binary_parameters() returns the orbital radii \(r_\odot\), \(r_p\) and angular velocity \(\omega\) of the binary from Kepler's third law. binary_positions(t, r_sun, r_planet, omega) evaluates the Cartesian coordinates of both binary members at any time \(t\); it is vectorised and accepts scalar or array arguments, allowing the entire output trajectory to be reconstructed in a single call after integration. mercpert_rhs(t, state, ...) is the right-hand-side function passed to solve_ivp: it unpacks the state vector \((x, y, v_x, v_y)\), computes the binary positions at the current internal time \(t\), evaluates the gravitational acceleration from both bodies, and returns a plain np.ndarray([v_x, v_y, a_x, a_y]) — the format required by SciPy.

driver_mercpert.py
Integration Driver

Defines the MercPertResult dataclass (lists of time and positions for all three bodies) and the function run_mercpert(). The driver assembles the initial state vector, builds the output time grid t_eval = np.arange(0, t_max, dt_output), and calls scipy.integrate.solve_ivp with method='RK45'. After the solver returns, the binary positions are reconstructed analytically at every output time point by a single vectorised call to binary_positions(sol.t, ...) — no binary positions are tracked inside the integration loop. The solver arrays and reconstructed binary positions are packed into a MercPertResult and returned.

plot_mercpert.py
Visualisation

Provides plot_orbits(result, title), which accepts a MercPertResult and renders a 6 × 6 inch matplotlib figure showing the trajectories of all three bodies in the \(xy\)-plane with equal aspect ratio: the Sun in red, the binary companion in green, and Mercury in blue. Initial positions are marked with filled circles. Axes use scientific notation and a light grid is included.

Three-Body Setup

The system contains three gravitationally interacting bodies. The Sun and Planet move on analytically prescribed circular orbits about their common barycentre; Mercury is a Newtonian test particle whose trajectory is integrated numerically.

  • The Sun — central star of mass \(M_\odot\) in solar-mass units, orbiting the barycentre at radius \(r_\odot\).
  • The Planet — massive companion of mass \(M_p\) in solar-mass units, orbiting the barycentre at radius \(r_p\) on the opposite side.
  • Mercury — test particle with negligible mass; exerts no gravitational force on either binary member.
Centre of mass convention: The binary barycentre is fixed at the origin. At \(t = 0\) the Sun lies on the negative \(x\)-axis and the Planet on the positive \(x\)-axis (consistent with the original Schutz Java convention). Both orbit counter-clockwise with angular velocity \(\omega\). Mercury's initial position and velocity are given relative to this same origin.

Binary Motion

Given the binary separation \(d\) and masses \(M_\odot\), \(M_p\) in solar-mass units, the orbital angular velocity follows from Kepler's third law:

(1) \[ \omega \;=\; \sqrt{\frac{G\,(M_\odot + M_p)^{\mathrm{SI}}}{d^3}} \]

The centre-of-mass condition gives each body's orbital radius:

(2) \[ r_\odot \;=\; \frac{M_p^{\mathrm{SI}}}{M_\odot^{\mathrm{SI}} + M_p^{\mathrm{SI}}}\,d, \qquad r_p \;=\; d - r_\odot \]

These are computed once by binary_parameters() in physics_mercpert.py. The Cartesian positions of both binary members at time \(t\) are then given by:

(3) \[ x_\odot(t) = -r_\odot\cos\omega t, \quad y_\odot(t) = -r_\odot\sin\omega t \]
(4) \[ x_p(t) = +r_p\cos\omega t, \quad y_p(t) = +r_p\sin\omega t \]

These expressions are evaluated by binary_positions(t, ...), which is vectorised so that the entire output trajectory can be reconstructed after integration with a single call over the full array sol.t.

Mercury's Equation of Motion

Let \(\mathbf{r} = (x, y)\) be Mercury's position. Define the displacement vectors pointing from each binary member toward Mercury:

(5) \[ \boldsymbol{\delta}_\odot = \mathbf{r} - \mathbf{r}_\odot(t), \qquad \boldsymbol{\delta}_p = \mathbf{r} - \mathbf{r}_p(t) \]

with scalar separations \(s_\odot = |\boldsymbol{\delta}_\odot|\), \(s_p = |\boldsymbol{\delta}_p|\). Mercury's total Newtonian acceleration is:

(6) \[ \ddot{\mathbf{r}} \;=\; -G\!\left( \frac{M_\odot^{\mathrm{SI}}\,\boldsymbol{\delta}_\odot}{s_\odot^3} + \frac{M_p^{\mathrm{SI}}\,\boldsymbol{\delta}_p}{s_p^3} \right) \]

This is implemented component-by-component in mercpert_rhs(), which returns the four-component state derivative:

(7) \[ \frac{d}{dt} \begin{pmatrix} x \\ y \\ v_x \\ v_y \end{pmatrix} \;=\; \begin{pmatrix} v_x \\ v_y \\ a_x \\ a_y \end{pmatrix} \]

where \((a_x, a_y)\) are the components of equation (6). This vector is returned as a plain np.ndarray so that solve_ivp can call it directly at every internal stage evaluation.

Non-autonomous ODE: Because the binary positions \(\mathbf{r}_\odot(t)\) and \(\mathbf{r}_p(t)\) rotate with angular velocity \(\omega\), the acceleration at position \(\mathbf{r}\) depends on both \(\mathbf{r}\) and the current time \(t\). Mercury's equation of motion is therefore non-autonomous. The RK45 integrator handles this transparently — each internal stage evaluation calls mercpert_rhs(t_stage, state_stage, ...) with the correct intermediate time — but it means the binary positions must be re-evaluated at every stage, not just at the start of each step.

Initial Conditions

Mercury's starting state is specified by four values in main.py: position (x_init_merc, y_init_merc) and velocity (vx_init_merc, vy_init_merc), all in SI units (metres and m s−1).

A useful reference is the circular orbital speed around the Sun alone at distance \(r_0\) from the barycentre:

(8) \[ v_c \;=\; \sqrt{\frac{G\,M_\odot^{\mathrm{SI}}}{r_0}} \]

Setting vy_init_merc close to \(v_c\) with y_init_merc = vx_init_merc = 0 produces a baseline orbit that would be circular in the absence of the binary companion. Deviations from circular motion make perturbation effects easier to see.

Default initial conditions: The default vy_init_merc = 30,000 m s−1 at \(r_0 = 0.5\,\mathrm{AU}\). The corresponding circular speed from equation (8) is approximately 37,600 m s−1, so the default orbit is deliberately sub-circular — it traces an ellipse rather than a circle, making perturbation-induced precession and distortion clearly visible.

RK45 Method

The Runge–Kutta–Fehlberg 4(5) method — commonly called RK45 — is an embedded pair of explicit Runge–Kutta schemes of order 4 and order 5 that share the same function evaluations. SciPy's implementation uses the Dormand–Prince tableau, which requires six stage evaluations per accepted step (with a seventh for the next step's predictor via FSAL — First Same As Last).

Stage Evaluations

At each step of width \(h\), the internal stage values \(\mathbf{k}_1, \ldots, \mathbf{k}_6\) are:

(9) \[ \mathbf{k}_i \;=\; \mathbf{f}\!\left( t_n + c_i\,h,\; \mathbf{y}_n + h\sum_{j=1}^{i-1} a_{ij}\,\mathbf{k}_j \right), \quad i = 1, \ldots, 6 \]

where \(\mathbf{f}\) is mercpert_rhs, and the coefficients \(c_i\) and \(a_{ij}\) are fixed by the Dormand–Prince Butcher tableau. Note that the internal time at stage \(i\) is \(t_n + c_i h\), not simply \(t_n\) — this correctly accounts for the time-varying binary positions in the non-autonomous ODE.

Fifth-Order Solution (Accepted State)

(10) \[ \mathbf{y}_{n+1} \;=\; \mathbf{y}_n + h\sum_{i=1}^{6} b_i\,\mathbf{k}_i \qquad \text{(5th order, used to advance)} \]

Fourth-Order Companion (Error Estimate Only)

(11) \[ \hat{\mathbf{y}}_{n+1} \;=\; \mathbf{y}_n + h\sum_{i=1}^{6} b_i^*\,\mathbf{k}_i \qquad \text{(4th order, used only to estimate error)} \]

Local Error Estimate

The per-component error estimate is the difference between the two solutions, computed for free from stage values already available:

(12) \[ \mathbf{e}_{n+1} \;=\; \mathbf{y}_{n+1} - \hat{\mathbf{y}}_{n+1} \;=\; h\sum_{i=1}^{6}(b_i - b_i^*)\,\mathbf{k}_i \]

This estimate drives the step-size controller described in the next section.

State Vector

The four-component state vector integrated by solve_ivp is assembled in driver_mercpert.py as:

(13) \[ \mathbf{y}_0 \;=\; \begin{pmatrix} x^{(0)} \\ y^{(0)} \\ v_x^{(0)} \\ v_y^{(0)} \end{pmatrix} \;=\; \begin{pmatrix} \texttt{x\_init\_merc} \\ \texttt{y\_init\_merc} \\ \texttt{vx\_init\_merc} \\ \texttt{vy\_init\_merc} \end{pmatrix} \]

After integration, Mercury's trajectory is extracted directly from the solver output: 

x_merc = sol.y[0]   # Mercury x-position  [m]
y_merc = sol.y[1]   # Mercury y-position  [m]
# sol.y[2], sol.y[3] hold vx, vy — available but not stored in MercPertResult

The binary positions are not tracked inside the integration loop. Instead, after solve_ivp returns, the driver reconstructs them analytically at every output time point in a single vectorised call: 

xs, ys, xp, yp = binary_positions(sol.t, r_sun, r_planet, omega)

This is both cheaper and more accurate than recording binary positions inside the loop, because it uses the exact analytical expressions for the circular orbit rather than whatever the integrator happened to have evaluated at each internal stage.

Error Control & Output Sampling

Tolerance Criterion

SciPy's RK45 implementation controls the step size so that the estimated local error in every state component satisfies a mixed absolute-relative criterion:

(14) \[ |e_i| \;\le\; \mathrm{atol} + \mathrm{rtol}\cdot|y_i| \qquad \text{for } i = 1, 2, 3, 4 \]

If this condition is satisfied for all four components the step is accepted. Otherwise the step is rejected, the step size is reduced, and the step is retried — automatically, without any intervention from the user.

Step-Size Growth Rule

After each accepted step, the solver proposes a new step size using the standard PI controller:

(15) \[ h_{\mathrm{new}} \;=\; h \cdot \min\!\left( 10,\; \max\!\left(0.2,\; 0.9 \cdot \left(\frac{\mathrm{tol}}{\|\mathbf{e}\|}\right)^{1/5} \right) \right) \]

This produces smooth, efficient step-size variation: steps contract near periapsis or during close encounters with the binary companion, and expand again when the trajectory becomes gentle.

Output Sampling vs. Integration Steps

The solver's internal step sizes are chosen purely to satisfy the error tolerances — they have no relationship to the output interval dt_output. The output grid is passed to solve_ivp as the t_eval array: 

t_eval = np.arange(0.0, t_max, dt_output)

SciPy evaluates the dense-output polynomial at each requested output time without any extra calls to mercpert_rhs. Changing dt_output therefore has no effect on integration accuracy or speed — it only controls how many trajectory points are stored and plotted.

Tip: If only the orbit shape matters and not fine temporal detail, a large dt_output (e.g. one binary orbital period) keeps memory usage small without sacrificing integration accuracy. If you want to resolve fast features such as close-encounter trajectories, shrink dt_output — but tighten rtol and atol as well, because the density of output points does not itself improve the underlying accuracy.

Suggested Scenarios

The following starting points, drawn from Schutz's original help file and adapted for the RK45 parameter convention, illustrate the range of behaviour the program can exhibit. In all cases the default binary separation 0.7 AU and tolerances rtol = atol = 1e-9 are recommended as a starting point; only the parameters listed need to be changed.

Gentle perturbation

Small companion mass produces slow precession of a nearly Keplerian ellipse.

m_planet_solar = 0.001
vy_init_merc = 37600 (near-circular)
t_max = 5e9

Strong perturbation

Increasing companion mass deforms the orbit significantly within a few revolutions.

m_planet_solar = 0.3
vy_init_merc = 30000
t_max = 2e8

Orbit flip (retrograde)

A sub-circular orbit can flip to retrograde after a close encounter with the companion.

m_planet_solar = 0.5
x_init_merc = 0.4 * AU
vy_init_merc = 20000
t_max = 5e8

Ejection

A close encounter can impart enough energy to eject Mercury from the system entirely.

m_planet_solar = 0.4
x_init_merc = 0.72 * AU
vy_init_merc = 25000
t_max = 3e8

Equal-mass binary

Setting m_sun_solar = m_planet_solar = 0.5 makes the two attractors identical; Mercury orbits a symmetric double star.

m_sun_solar = 0.5
m_planet_solar = 0.5
x_init_merc = 1.5 * AU
vy_init_merc = 20000

Jupiter sweep

Vary m_planet_solar from 0.001 to 0.3 in steps and observe the transition from regular to chaotic orbit.

Run multiple instances of main.py with different values and overlay the plots.
